# 自定义词你画我猜终端协作文档

## 目标

先把 `game` 页房主端恢复可用，再继续推进“每轮由画手现场输入自定义词条”的完整闭环。

## 当前功能定义

先统一口径，避免后续又改回旧方案：

- 创建房间时不预存词库。
- 每一轮开始后，当前画手先进入 `choosing` 状态。
- 画手在游戏页输入本轮词条。
- 提交成功后进入 `drawing` 状态并启动倒计时。
- 其他玩家在聊天区猜词。

后续所有终端改动，都默认基于这个定义继续推进。

## 当前已完成的部分

这些不需要再作为优先任务重复做：

- 创建页已经清掉密码和题型选择假功能。
- `startRound` 已改成“先建空词回合，再等画手提交词”。
- `submitWord` 已经存在。
- 游戏页已经能区分出题阶段和绘画阶段。
- 聊天区已经能展示猜词消息和猜中消息。
- 结束页的明显变量错误已经修掉。

## 本轮测试确认的最高优先级问题

当前先解决这两件事，否则无法继续联调：

1. 房主端看不到“开始游戏”按钮。
2. 房主端看不到玩家头像列表。

补充优先问题：

1. `game` 页状态同步整体不稳定。
2. 开始游戏后，画手输入词条区仍有不出现的情况。
3. 房间密码功能不在范围内，不再保留任何待办。

## 推荐执行顺序

### 第 1 阶段：修房主端等待区可用性（必须先做）

目标：房主进入房间后稳定看到玩家头像和“开始游戏”按钮。

任务：

1. 检查并固定等待区渲染条件：
   - `gameStatus === 'waiting'`
   - `isOwner === true`
   - `players.length > 0`
2. 给房间实时监听加兜底：
   - watcher 建立后主动拉一次房间快照
   - watcher 异常时触发补拉
3. 在页面上增加最小调试信息（仅开发态）：
   - `gameStatus`
   - `isOwner`
   - `players.length`
4. 确认房主身份来源稳定：
   - `ownerOpenid`
   - 当前 `openid`
   - 两者比较结果

验收：

- 房主端稳定显示“开始游戏”按钮。
- 房主端稳定显示玩家头像列表。
- 新玩家进入时房主端实时刷新。

### 第 2 阶段：修 `game` 页同步主链路

目标：房主端和玩家端看到同一份房间状态。

任务：

1. 检查 `onLoad -> enterRoom -> roomWatcher -> onRoomSnapshot` 顺序。
2. 明确关键状态单一来源：
   - `gameStatus`
   - `players`
   - `isOwner`
3. 修正页面内局部 `setData` 与快照数据冲突问题。
4. 保留“房主不参与准备”规则，不回退。

验收：

- 房主端和玩家端的成员列表一致。
- 房主端与玩家端的等待/开始状态一致。

### 第 3 阶段：修复画手输入词条界面

目标：开始游戏后，当前画手一定能看到词条输入区，其他玩家看到等待提示。

任务：

1. 检查 `startGame -> startRound -> room.currentRoundId -> round watcher -> roundStatus` 这条链路。
2. 不要只依赖“异步 watcher 恰好更新成功”才显示输入区。
3. 必要时增加兜底逻辑：
   - `startRound` 成功后主动拉取当前 round
   - 或在房间状态变化后补一次回合数据同步
4. 明确“输入词条界面”是页面区块还是弹层。
   - 如果继续用页面区块，需要保证显示条件稳定。
   - 如果要改成弹层，也要写清楚状态来源。

验收：

- 开始游戏后，画手一定进入出题状态。
- 其他玩家一定看到“等待画手出题”。

### 第 4 阶段：补回合闭环

目标：让一轮能完整结束，而不是停在倒计时归零。

任务：

1. 明确由云函数负责结束回合，不依赖前端本地计时器做业务推进。
2. 增加回合结束触发条件：
   - 时间到
   - 或所有非画手玩家已猜中
3. 补一个明确的推进函数或重构 `endRound`，负责：
   - 写 `round.endTime`
   - 写 `round.status`
   - 结算本轮分数
   - 判断是否进入下一轮
   - 判断是否结束整局
4. 如果进入下一轮：
   - 选出下一位画手
   - 调 `startRound`
5. 如果结束整局：
   - 调 `endGame`

验收：

- 不需要人工点击额外按钮，一局能从第一轮推进到下一轮。

### 第 5 阶段：补猜中后的玩家状态与分数

目标：让猜中行为不只是聊天消息，而是真正驱动状态变化。

任务：

1. 修改 `sendGuess`：
   - 猜对后更新 `room.players` 中对应玩家的 `hasGuessed`
   - 防止同一玩家重复猜中重复加分
2. 明确计分规则：
   - 猜中者得分
   - 画手是否得分
   - 是否按剩余时间加权
3. 在每轮开始时重置所有非画手的 `hasGuessed`
4. 前端继续沿用 `players` 渲染状态，不另起一套临时状态

验收：

- 玩家列表中的“已猜中”图标能真实变化。
- 同一轮不能重复刷分。

### 第 6 阶段：补整局结束展示链路

目标：`finished` 状态时，前端能拿到稳定的最终分数。

任务：

1. 二选一明确最终分数来源：
   - 方案 A：`endRound` / `endGame` 同步把 `totalScore` 写回 `room`
   - 方案 B：前端在 `finished` 后主动拉 `games`
2. 修正 `pages/game/game.js` 中对 `doc.totalScore` 的依赖，改成和后端一致的真实来源。
3. 确保 `finalScores` 至少在空分数时也能稳定展示，不要因为 `undefined` 变成空白弹窗。

验收：

- 游戏结束后排行榜稳定可见，昵称和分数正确。

### 第 7 阶段：收敛入房逻辑

目标：房间加入流程只保留一套可信实现。

任务：

1. 确定保留 `enterRoom` 还是 `joinRoom`。
2. 删除另一套重复加入逻辑，避免首页和游戏页各维护一份。
3. 统一玩家结构：
   - `openid`
   - `nickName`
   - `avatarUrl`
   - `score`
   - `isReady`
   - `hasGuessed`

验收：

- 同一个玩家加入房间只有一个正式入口。

### 第 8 阶段：清理和增强

目标：降低维护噪音，提升后续可改性。

任务：

1. 删除未使用代码：
   - `throttle`
   - 未使用的 `_ = db.command`
2. 修正直接修改 `this.data` 的写法。
3. 给房间码创建增加唯一性校验。
4. 补最少量日志，覆盖：
   - 开始游戏
   - 开始回合
   - 提交词条
   - 猜中
   - 结束回合
   - 结束整局

验收：

- 代码里没有明显的死代码和重复入口。
